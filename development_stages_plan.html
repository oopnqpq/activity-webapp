四階段開發計畫（最終版）

原有項目 新增 / 調整項目 技術風險點
Stage 1 基礎建設 — 資料結構 + 驗證碼 + Email 3–4 天
Google Sheets 資料表設計（驗證碼、分組、各關卡狀態欄位） Apps Script API 骨架 + Properties token 設定 驗證碼批次產生（Sheets RANDBETWEEN 公式） GitHub Pages 部署 + HTTPS QR Code 產生（報到QR / 簽退QR 各一套，qrcode.js） Google Forms 報名資料 → Sheets 主資料庫：Apps Script 一鍵同步（欄位對應 + 去重保護） Email 批次寄送（前台網址 + 驗證碼） localStorage + 12 小時過期 session 邏輯

⚠ Gmail 100封/日：Day1 寄 100 封、Day2 寄剩餘 50 封，活動前完成。

開發：3–4 天
Stage 2 後台 — 工作人員三大功能 + 儀表板 5–6 天
密碼登入（Properties token 驗證） jsQR 掃碼通用元件 報到：掃QR → 顯示組別+個人資訊 → 防呆確認 → 更新 簽退：掃QR → 確認報到+使命均完成 → 兌換禮品防呆 → 更新 核銷管理：搜尋 → 手動標記 → 防呆 → 更新 儀表板：各關卡即時進度 + 參與者名單搜尋 LockService 並發保護（所有寫入操作必加）

⚠ LockService 是整個系統最關鍵的技術點，Stage 2 第一天優先驗證：模擬 20 支並發請求，確認 waitLock(10000) 能正確排隊不漏寫。證書設計稿需在本階段結束前取得（低解析度稿供 Stage 3 測試用）。

開發：5–6 天
Stage 3 前台 — 參與者完整流程（最複雜） 8–10 天
登入頁：姓名+驗證碼 → Apps Script 驗證 → 返回個人資料 localStorage 儲存 session（12小時過期，熄屏返回保持登入） 個人資訊：姓名、組別、報到QR Code 顯示 安全提醒：頁籤切換 + 圖片點擊放大 路線頁：獨立靜態頁面、地圖圖片顯示 使命①：Google Forms 問卷（外開新分頁 + 手動勾選確認，零開發量） 使命②：照片選取（1張）→ base64 存瀏覽器記憶體 使命③：signature_pad.js 手寫簽名 → canvas base64 使命④：jsPDF — 證書PNG底圖 + 姓名文字 + 簽名 + 活動照片，純前端合成 使命⑤：PDF 取得方式二選一 — 下載至手機（Blob API，0 API）或輸入 Email 寄送（Apps Script Gmail，含於 ⑥ 批次 API 送出） 使命⑥：單一批次 API 送出（照片 base64 + PDF base64 + token 合併一次呼叫） Apps Script 批次收到後：建個人 Drive 資料夾 → 上傳照片 → 上傳 PDF → 更新 Sheets；若 ⑤ 選擇 Email 寄送，同次呼叫一併觸發 Gmail 發送 任務完成後才顯示簽退 QR Code 未完成時提示尚缺項目清單 使命解鎖邏輯：① 與 ②～⑤ 並行獨立，⑥ 送出前統一 Checklist 驗證雙線均完成（雙 boolean flag，非 Gated Unlock） 指數退避重試（Exponential Backoff，批次 API 失敗最多重試 3 次）

⚠ 四個技術風險需在 Stage 3 第一天提前測試：① jsPDF CJK字體檔 ~3MB，測試手機冷載入時間是否可接受；② 照片+PDF base64 合併一次送出，4G 環境約 8–15 秒，需明顯 loading 動畫；③ Apps Script 批次處理含 Drive 寫入，執行時間需控制在 6 分鐘以內（理論上單次 10–20 秒，遠低於上限）；④ ⑤ Email 寄送路徑：Gmail 100封/日配額需評估，若多數人選擇寄送有超限風險，備案為改用 mailto: 連結開啟使用者自己的信箱用戶端。

開發：8–10 天
